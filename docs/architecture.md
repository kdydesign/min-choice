# 아키텍처 문서

## 1. 문서 목적

이 문서는 베베 초이스의 **시스템 구조와 데이터 흐름을 설명하는 기술 설계 문서**입니다.  
제품 범위는 [./product-spec.md](./product-spec.md), 구현 규칙은 [../AGENTS.md](../AGENTS.md)를 따르며, 이 문서는 그 사이의 **구조 설명**을 담당합니다.

## 2. 시스템 개요

베베 초이스는 12개월 전후 아이를 위한 하루 3끼 식단 추천 앱입니다.  
사용자는 아이 프로필을 기준으로 아침, 점심, 저녁 재료를 입력하고 식단을 생성합니다.

핵심 구조는 아래와 같습니다.

- 프론트엔드: Vite + React + TypeScript 기반 PWA
- 인증/데이터: Supabase Auth + Postgres
- 서버 기능: Supabase Edge Functions
- 배포: Vercel
- AI: 서버사이드 OpenAI 호출

중요한 원칙:

- 메뉴 선택은 규칙 기반으로 처리합니다.
- AI는 메뉴를 고르지 않고 자연어 결과만 생성합니다.
- AI 호출은 서버 함수에서만 수행합니다.
- 알레르기 재료는 완전 제외합니다.
- source of truth는 Supabase DB입니다.

## 3. 런타임 구성

### 3.1 Vercel

Vercel은 프론트엔드를 호스팅합니다.

역할:

- 정적 프론트 배포
- SPA 라우팅 제공
- PWA 자산 제공
- 공개 HTTPS URL 제공

### 3.2 Supabase Auth

Supabase Auth는 사용자 인증을 담당합니다.

역할:

- Google 로그인
- 익명 체험 세션
- 세션 복원
- 익명 사용자와 로그인 사용자 연결

### 3.3 Postgres

Supabase Postgres는 애플리케이션의 핵심 데이터를 저장합니다.

주요 저장 대상:

- 아이 프로필
- 재료 기준 데이터
- 메뉴 기준 데이터
- 끼니 입력 재료
- 하루 식단 결과
- 끼니별 상세 결과
- AI 생성 로그
- 익명 사용자 연결 정보

### 3.4 Edge Functions

Supabase Edge Functions는 서버사이드 비즈니스 로직을 담당합니다.

주요 함수:

- `normalize-ingredients`
- `generate-meal-plan`
- `link-anonymous-user`

역할:

- 재료 정규화
- 서버사이드 AI 호출
- AI 응답 검증
- fallback 처리
- 익명 데이터 계정 연결

## 4. 프론트엔드 구조

### 4.1 `src/app`

앱 엔트리, 라우터, 전역 프로바이더를 담당합니다.

예:

- React Query Provider
- Auth Provider
- Router 설정

### 4.2 `src/pages`

라우트 단위 화면을 담당합니다.

예:

- 로그인 화면
- 오늘 식단 화면
- 최근 식단 화면
- 아이 프로필 화면

### 4.3 `src/components`

공용 UI 컴포넌트를 담당합니다.

예:

- 버튼
- 카드
- 태그 입력
- 모바일 선택 컴포넌트
- 상태 컴포넌트
- 공용 레이아웃

### 4.4 `src/features`

도메인별 기능을 담당합니다.

포함 내용:

- API 호출
- 타입
- 훅
- 검증 로직
- 도메인 유틸

### 4.5 `src/lib`

외부 클라이언트와 공용 유틸을 담당합니다.

예:

- Supabase 클라이언트
- 환경 변수 처리
- 범용 유틸

### 4.6 `src/services`

브라우저 저장소와 보조 서비스를 담당합니다.

예:

- localStorage 접근
- draft 저장
- 세션 보조 처리

### 4.7 `src/store`

Zustand 기반의 가벼운 클라이언트 상태를 담당합니다.

예:

- 선택한 아이
- 선택한 식단
- 화면 보조 상태

## 5. 주요 도메인

### 5.1 `auth`

책임:

- Google 로그인
- 익명 체험
- 세션 복원
- 익명 계정 연결 트리거

### 5.2 `children`

책임:

- 아이 프로필 CRUD
- 개월 수와 생년월일 동기화
- 알레르기 재료 관리
- 프로필 등록/수정 폼의 모바일 입력 안정성 유지

### 5.3 `ingredients`

책임:

- 재료 정규화
- 입력 재료 표준화
- 알레르기 입력 경고 보조

### 5.4 `meal-plans`

책임:

- 오늘 식단 입력
- 식단 생성 요청
- 식단 결과 조회
- 최근 식단 이력 조회
- 결과 재생성

### 5.5 `menus`

책임:

- 메뉴 기준 데이터 참조
- 규칙 기반 후보 선택에 필요한 메타데이터 제공

## 6. 인증 흐름

### 6.1 Google 로그인

1. 사용자가 Google 로그인 버튼을 누릅니다.
2. 프론트는 Supabase OAuth를 시작합니다.
3. 인증 완료 후 앱 루트로 복귀합니다.
4. 세션이 복원되고 사용자 데이터 조회가 시작됩니다.

### 6.2 익명 체험

1. 사용자가 익명으로 계속하기를 선택합니다.
2. Supabase 익명 세션을 생성하거나 기존 익명 세션을 재사용합니다.
3. 사용자 데이터는 익명 사용자 기준으로 저장됩니다.

### 6.3 익명 데이터 복원

1. 같은 브라우저에서 다시 진입합니다.
2. 기존 익명 세션 또는 복원 가능한 익명 상태를 재사용합니다.
3. 이전 아이 프로필과 식단 데이터가 다시 조회됩니다.

### 6.4 익명 데이터 계정 연결

1. 사용자가 익명 상태에서 데이터를 생성합니다.
2. 이후 Google 로그인합니다.
3. `link-anonymous-user` 함수가 익명 데이터와 로그인 계정을 연결합니다.
4. 이후 같은 데이터를 로그인 사용자 기준으로 계속 사용합니다.

## 7. 식단 생성 데이터 흐름

### 7.1 입력

사용자는 선택된 아이를 기준으로 아침, 점심, 저녁 재료를 입력합니다.

입력 특징:

- 태그 기반 입력
- 재료별 독립 입력
- 알레르기 재료 즉시 경고

### 7.2 재료 정규화

입력 재료는 먼저 표준 키로 정규화됩니다.

역할:

- 오탈자와 별칭 보정
- 표준 ingredient key 매핑
- 알레르기 재료 확인

### 7.3 규칙 기반 후보 선택

정규화된 재료를 기준으로 규칙 기반 추천 엔진이 끼니별 후보와 fallback 메뉴를 계산합니다.

역할:

- 메뉴 후보 계산
- fallback 메뉴 결정
- 자동 보완 재료 계산
- 후보 메타 제공

중요:

- 이 단계는 AI 선택을 위한 guardrail과 fallback을 제공합니다.
- 최종 메뉴 선택의 1차 책임은 서버사이드 AI에 있습니다.

### 7.4 AI 메뉴 선택 + 자연어 생성

서버 함수가 끼니별로 OpenAI를 호출해 메뉴 선택과 자연어 결과를 함께 생성합니다.

AI가 담당하는 것:

- 메뉴명 선택
- 조리 스타일 선택
- 사용 재료 / 부족 재료 / 대체재 제안
- 추천 문구
- 부족 재료 설명
- 3줄 요약 조리법
- 5~8단계 상세 조리법
- 주의사항
- 영양/조리시간 추정

중요:

- AI 호출은 서버 함수에서만 수행합니다.
- 클라이언트에서 OpenAI를 직접 호출하지 않습니다.
- 끼니별 최대 2회까지 재시도하고, 하루 전체 최대 6회까지 호출할 수 있습니다.

### 7.5 검증

AI 응답은 저장 전에 검증합니다.

검증 기준:

- 구조화된 JSON 파싱 가능
- 월령 적합성
- 알레르기 재료 미포함
- 위험 표현 없음
- 최근 3일 이력 / 현재 요청 earlier meal 중복 회피
- `ingredient_first` 허용 재료 범위 준수
- `auto_recommend` 표준 재료 집합 준수
- 영양/조리시간 허용 오차 검증

### 7.6 저장

검증된 결과는 DB에 저장합니다.

저장 테이블:

- `meal_plans`
- `meal_inputs`
- `meal_plan_items`
- 필요 시 `ai_generation_logs`

원칙:

- source of truth는 Supabase DB입니다.
- 로컬 저장은 결과의 기준이 아닙니다.
- `selectionSource`, `nutritionSource`는 `result_payload_json`으로 함께 저장합니다.

### 7.7 결과 표시

저장된 결과를 기준으로 오늘 식단 결과 화면을 보여줍니다.

표시 정보:

- 메뉴명
- 추천 문구
- 사용 재료
- 부족 재료
- 대체 가능한 재료
- 주의사항
- 3줄 조리법
- fallback 여부

## 8. 상태 관리 원칙

### 8.1 React Query

서버 상태는 React Query로 관리합니다.

대상:

- 아이 프로필 목록
- 최신 식단 결과
- 최근 식단 이력

원칙:

- 서버 데이터는 React Query 기준으로 조회/갱신합니다.
- 서버 응답을 별도 전역 상태의 source of truth로 복제하지 않습니다.

### 8.2 Zustand

가벼운 클라이언트 상태는 Zustand로 관리합니다.

대상:

- 선택한 아이
- 선택한 식단
- UI 보조 상태

원칙:

- 서버 데이터 캐시 전체를 Zustand에 넣지 않습니다.

### 8.3 localStorage

localStorage는 제한적으로만 사용합니다.

허용 범위:

- 선택한 아이
- 작성 중인 식단 draft
- 익명 세션 복원에 필요한 최소 키

원칙:

- localStorage는 편의 기능과 세션 복원만 담당합니다.
- source of truth는 Supabase DB입니다.

## 9. 프론트엔드 UI 런타임 기준

### 9.1 모바일 선택 UI

`MobileSelect`는 개월 수와 생년월일 선택에서 사용하는 공통 컴포넌트입니다.

책임:

- 선택값 표시
- placeholder 표시
- 옵션 팝업 열림/닫힘 상태 관리
- 옵션 목록 스크롤
- ESC / 외부 클릭 / 닫기 버튼 / 옵션 선택 cleanup
- 팝업 open 중 배경 스크롤 잠금

원칙:

- 특정 도메인의 날짜 계산 로직을 포함하지 않습니다.
- 값 타입은 `string | number`를 유지합니다.
- 페이지별 중복 select UI를 만들지 않습니다.

### 9.2 프로필 화면 viewport 처리

프로필 화면은 고정 헤더, 내부 스크롤 컨테이너, 고정 하단 탭을 함께 사용합니다.

구현 기준:

- `profile-page-active` class로 프로필 화면에서만 body 외부 스크롤을 잠급니다.
- 키보드 open 상태는 `visualViewport`로 감지하되, 페이지 높이를 직접 줄이지 않습니다.
- 키보드 open 중 하단 탭은 숨기고 pointer event를 차단합니다.
- 키보드 close 후 내부 `.profile-selection-layout`의 scrollTop을 유효 범위로 보정하고 body scroll을 `0,0`으로 복원합니다.

주의:

- iOS Chrome도 WebKit 기반이므로 Safari와 동일한 viewport 보정 경로를 사용합니다.
- PWA standalone 전용 처리는 별도 범위입니다.

## 10. 보안 원칙

### 10.1 RLS

RLS를 유지합니다.

원칙:

- 사용자는 자신의 데이터만 접근해야 합니다.
- `menus`, `ingredients`는 읽기 전용으로 둡니다.
- AI 로그는 일반 사용자에게 직접 노출하지 않습니다.

### 10.2 시크릿 분리

시크릿은 서버에서만 관리합니다.

예:

- `OPENAI_API_KEY`
- `service_role`

원칙:

- 프론트에는 브라우저용 공개 키만 둡니다.
- 서버용 시크릿은 Edge Function 시크릿으로 관리합니다.

### 10.3 클라이언트에서 하면 안 되는 것

- OpenAI 직접 호출
- 서버 시크릿 보관
- 알레르기 규칙 우회
- DB source of truth를 localStorage로 대체
