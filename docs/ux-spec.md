# 베베 초이스 UX 스펙

## 문서 목적

이 문서는 베베 초이스의 **최종 UX / UI 기준 문서**입니다.  
제품 범위는 [product-spec.md](./product-spec.md), 구현 규칙은 [../AGENTS.md](../AGENTS.md)를 참고하되, 화면 구조와 UX 의사결정은 이 문서를 우선합니다.

## 1. UX 목표

사용자는 **1분 이내에** 아래 흐름을 끝낼 수 있어야 합니다.

1. 아이를 선택한다.
2. 끼니별 재료를 입력한다.
3. 하루 식단을 생성한다.
4. 결과를 확인한다.

앱은 아래 특성을 가져야 합니다.

- 모바일 퍼스트
- 카드형 구조
- 낮은 인지 부담
- 따뜻하고 안정적인 분위기
- 알레르기 경고 우선

## 2. 디자인 원칙

### 2.1 모바일 우선

- 모든 화면은 모바일을 기준으로 설계합니다.
- 한 손 사용이 쉬워야 합니다.
- 주요 CTA는 화면 하단 가까이에 두는 것을 우선합니다.

### 2.2 명확함 우선

- 화면마다 사용자가 해야 할 일을 바로 이해할 수 있어야 합니다.
- 장식보다 정보 구조와 가독성을 우선합니다.
- 한 화면에 너무 많은 액션을 동시에 두지 않습니다.

### 2.3 안전 우선

- 알레르기 경고는 일반 정보보다 항상 우선합니다.
- 경고와 주의사항은 결과 카드 안에서도 눈에 잘 띄어야 합니다.

### 2.4 재사용 가능 구조

- 카드, 버튼, 입력, 경고 배너, 상태 컴포넌트는 반복 가능한 패턴으로 구성합니다.
- 새로운 화면을 만들 때도 기존 토큰과 컴포넌트를 재사용합니다.

## 3. 시각 방향

### 3.1 브랜드 톤

- 따뜻함
- 안정감
- 청결함
- 신뢰감
- 과하지 않은 귀여움

### 3.2 컬러 방향

- 부드러운 파스텔 계열
- 크림, 베이지, 연블루, 연그린, 연피치 중심
- 경고는 따뜻한 옐로우/오렌지 계열

### 3.3 형태

- 라운드 카드
- 큰 터치 영역
- 가벼운 그림자
- 과한 모션 금지

## 4. 정보 구조

앱의 핵심 화면은 아래 4개입니다.

- 로그인 화면
- 아이 프로필 화면
- 오늘 식단 입력/결과 화면
- 최근 식단 화면

하단 탭 또는 명확한 네비게이션으로 이동할 수 있어야 합니다.

## 5. 화면별 UX 기준

### 5.1 로그인 화면

목표:
- 빠르게 앱 진입
- Google 로그인과 익명 체험을 명확히 선택

구성:
- 중앙 로고 또는 브랜딩
- Google 로그인 버튼
- 익명으로 계속하기 버튼
- 불필요한 설명 최소화
- 하단 카피라이트

원칙:
- CTA 외 정보는 최소화
- 첫 화면의 진입 속도를 방해하지 않음

### 5.2 아이 프로필 화면

목표:
- 아이 선택
- 새 아이 등록
- 기존 아이 수정

구성:
- 상단 헤더
- 아이 카드 리스트
- `아이 추가하기` CTA
- 등록/수정 폼

폼 기준:
- 이름
- 개월 수
- 생년월일
- 알레르기 태그

입력 규칙:
- 개월 수와 생년월일은 자동 동기화
- 개월 수와 생년월일의 년/월/일 선택은 모바일 터치에 적합한 공통 선택 컴포넌트를 사용
- 선택 팝업은 화면 중앙에 표시하고, 열린 동안 배경 스크롤이 움직이지 않아야 함
- 선택 전 placeholder와 선택 후 값을 명확히 구분
- 월/연도 변경으로 선택된 일이 유효하지 않으면 해당 월의 마지막 일자로 보정
- 미래 생년월일은 선택 또는 저장 단계에서 방어
- 오류는 필드 가까이에 표시
- 저장/수정/삭제 액션은 명확히 구분
- 이름 입력 후 모바일 키보드가 닫혀도 하단 탭 위에 불필요한 공백이 남지 않아야 함
- 알레르기 태그 입력은 포커스 상태가 명확해야 하며, 태그 추가/삭제 동작을 유지해야 함

### 5.3 오늘 식단 입력 화면

이 화면은 제품에서 가장 중요합니다.

목표:
- 아이 기준으로 오늘 3끼 재료를 빠르게 입력
- 알레르기 충돌을 즉시 확인
- 하루 식단 생성으로 자연스럽게 이어짐

필수 구성:
- 선택된 아이 카드
- 아침 재료 입력 카드
- 점심 재료 입력 카드
- 저녁 재료 입력 카드
- 알레르기 안내 박스
- `하루 식단 생성하기` CTA

입력 UX 규칙:
- 태그 입력 기반
- 재료 추가/삭제가 쉬워야 함
- 알레르기 재료는 즉시 경고
- 재료가 바뀌면 이전 추천 결과는 숨김

### 5.4 오늘 식단 결과 화면

목표:
- 하루 3끼 결과를 한 번에 이해
- 필요하면 상세만 펼쳐서 확인

필수 구성:
- 상단 뒤로가기
- 결과 제목
- 끼니 카드 3개
- 조리 팁 카드
- `재료 다시 입력`
- `식단 다시 생성`

끼니 카드 필수 정보:
- 메뉴명
- 추천 문구
- 사용 재료
- 부족 재료
- 대체 가능한 재료
- 주의사항
- 조리법 3줄
- 필요 시 fallback 여부

상세 UX:
- 끼니 카드는 접기/펼치기를 지원
- 생성 중에는 토글과 액션이 모두 잠겨야 함

### 5.5 최근 식단 화면

목표:
- 저장된 식단을 빠르게 다시 찾기
- 오늘 / 최근 / 이전 흐름으로 쉽게 스캔

구성:
- 선택된 아이 기준 이력
- 날짜 그룹
- 카드 요약
- `불러오기` 액션

원칙:
- 긴 리스트보다 그룹화가 우선
- 선택된 식단과 상세 결과 연결이 명확해야 함

## 6. 공통 컴포넌트 UX 기준

### 6.1 ChildProfileCard

- 이름, 개월 수, 알레르기 태그를 보여줍니다.
- 선택 상태가 명확해야 합니다.

### 6.2 IngredientTagInput

- 태그 추가/삭제가 쉬워야 합니다.
- 입력 중 경고 태그는 시각적으로 구분합니다.
- 비활성 상태에서는 입력과 태그 삭제가 모두 막혀야 합니다.

### 6.3 MobileSelect

- 단순 HTML select 노출 대신 터치 친화적인 커스텀 선택 UI를 사용합니다.
- 트리거는 충분한 높이와 명확한 선택값 표시를 가져야 합니다.
- 옵션 팝업은 화면 중앙 모달 또는 모바일 친화 레이어로 표시합니다.
- 팝업이 열려 있는 동안 배경 스크롤은 잠그고, 옵션 목록 내부 스크롤만 허용합니다.
- 외부 클릭, 닫기 버튼, ESC, 옵션 선택으로 닫을 수 있어야 합니다.
- `aria-expanded`, `aria-haspopup`, `role="listbox"` 등 접근성 속성을 유지합니다.

### 6.4 WarningBanner

- 알레르기, 위험 요소, 주의사항을 노출합니다.
- 일반 정보 카드보다 시각적 우선순위가 높아야 합니다.

### 6.5 LoadingState / ErrorState / EmptyState

- 모든 주요 화면에서 공통 패턴을 유지합니다.
- 상태 메시지는 짧고 명확해야 합니다.

## 7. 로딩 및 상태 UX

### 7.1 오늘 식단 생성 상태

오늘 식단 생성은 **실제 단계 기반**으로 표시합니다.

- 1단계: 재료 이름 정리
- 2단계: 식단 생성
- 3단계: 저장

progress bar는 단계형으로만 반영합니다.

- 1단계 -> `20%`
- 2단계 -> `80%`
- 3단계 -> `100%`

중요:
- 이 bar는 바이트 단위 실제 퍼센트가 아닙니다.
- 실제 비동기 단계 상태를 반영하는 단계형 UI입니다.

### 7.2 생성 중 상호작용 차단

오늘 식단 생성 중에는 현재 화면의 입력과 버튼을 모두 막습니다.

입력 화면:
- 재료 입력
- 입력 초기화
- 프로필 이동
- 생성 버튼

결과 화면:
- 뒤로가기
- 재료 다시 입력
- 상세 토글
- 식단 다시 생성

## 8. 빈 상태 / 오류 상태

### 8.1 아이 없음

- 프로필이 하나도 없으면 프로필 화면으로 먼저 유도합니다.
- 빈 상태는 등록 CTA를 포함해야 합니다.

### 8.2 결과 없음

- 식단 결과가 없으면 재료 입력과 생성 CTA로 돌아가게 해야 합니다.

### 8.3 생성 실패

- 오류 메시지는 사용자가 이해할 수 있어야 합니다.
- 입력값은 유지하고 다시 시도할 수 있어야 합니다.

## 9. 접근성 기준

- 터치 영역은 충분히 커야 합니다.
- 텍스트 대비는 읽기 쉬워야 합니다.
- `aria-label`, `aria-busy`, `disabled` 상태를 적절히 제공합니다.
- 모션은 최소화하고, 상태 변화는 명확하게 전달합니다.
- 모바일 키보드, 브라우저 주소창, safe area 변화가 고정 헤더/하단 탭과 충돌하지 않아야 합니다.
- 모달 또는 선택 팝업이 열려 있는 동안 배경 스크롤이 사용자 의도와 다르게 움직이면 안 됩니다.

## 10. 구현 시 체크리스트

새 화면이나 컴포넌트를 만들 때는 아래를 먼저 확인합니다.

- 모바일 퍼스트인가
- 카드형 구조인가
- 디자인 토큰이 일관적인가
- 재사용 가능한 구조인가
- 알레르기 경고 UX가 우선되는가
- 생성 중 상태와 오류 상태가 정의되어 있는가
