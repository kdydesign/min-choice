# 베베 초이스 제품 스펙

## 문서 목적

이 문서는 베베 초이스의 **제품 범위, 사용자 흐름, 기능 요구사항, 비기능 요구사항**을 정의합니다.  
화면 설계는 [ux-spec.md](./ux-spec.md), 시스템 구조는 [architecture.md](./architecture.md), 구현 규칙은 [../AGENTS.md](../AGENTS.md)를 참고합니다.

## 1. 서비스 개요

### 1.1 제품명

베베 초이스 (Bebe Choice)

### 1.2 제품 목적

보호자가 집에 있는 재료를 끼니별로 입력하면, 12개월 전후 아이에게 적합한 하루 3끼 식단을 빠르게 추천하는 모바일 중심 앱을 제공합니다.

### 1.3 해결하려는 문제

- 집에 있는 재료로 무엇을 만들 수 있을지 빠르게 떠올리기 어렵다.
- 12개월 아이 기준으로 안전한 식단을 매일 구성하기 어렵다.
- 하루 3끼를 한 번에 조합하는 데 시간이 오래 걸린다.
- 알레르기 재료를 고려한 메뉴 선택이 부담된다.

## 2. 핵심 사용자

- 12개월 전후 아이를 키우는 부모
- 아이 식단을 직접 준비하는 보호자
- 둘 이상의 아이 프로필을 나누어 관리하려는 사용자
- 모바일에서 빠르게 식단을 만들고 싶은 사용자

## 3. 현재 제품 범위

### 3.1 포함 범위

- Google 로그인
- 익명 체험
- 익명 데이터의 Google 계정 연결
- 아이 프로필 생성, 수정, 삭제, 선택
- 아이별 알레르기 재료 관리
- 아침 / 점심 / 저녁 재료 입력
- 재료 정규화
- 규칙 기반 메뉴 후보 선택
- 하루 3끼 식단 생성
- AI 추천 문구 / 조리법 / 주의사항 생성
- 최근 식단 이력 조회
- 모바일 PWA 설치와 사용

### 3.2 현재 제외 범위

- 카카오 로그인 운영
- 주간 식단
- 장보기 연동
- 영양 성분 상세 분석
- 전문가 상담 기능

## 4. 핵심 사용자 흐름

### 4.1 첫 진입

1. 사용자가 앱에 진입한다.
2. Google 로그인 또는 익명 체험을 선택한다.
3. 아이 프로필이 없으면 프로필 화면으로 먼저 이동한다.
4. 아이 프로필을 등록한 뒤 오늘 식단 화면으로 이동한다.

### 4.2 오늘 식단 생성

1. 사용자가 아이를 기준으로 오늘 식단 화면에 진입한다.
2. 아침 / 점심 / 저녁 재료를 입력한다.
3. 알레르기 재료가 입력되면 즉시 경고를 확인한다.
4. `하루 식단 생성하기`를 누른다.
5. 시스템은 재료 정규화, 메뉴 후보 선택, AI 자연어 생성, 저장을 수행한다.
6. 사용자는 오늘 식단 결과 화면에서 3끼 결과를 확인한다.

### 4.3 재생성

1. 사용자가 재료를 수정하거나 결과 화면에서 다시 생성을 누른다.
2. 시스템은 현재 입력을 기준으로 식단을 다시 만든다.
3. 결과가 저장되면 최신 식단으로 갱신한다.

### 4.4 익명 사용 후 로그인

1. 사용자가 익명으로 프로필과 식단을 만든다.
2. 시작 화면으로 돌아가더라도 같은 브라우저에서는 데이터를 유지한다.
3. 이후 Google 로그인 시 익명 데이터를 계정에 연결한다.

## 5. 기능 요구사항

### 5.1 아이 프로필

- 프로필에는 아래 정보가 포함되어야 한다.
  - 이름
  - 개월 수
  - 생년월일
  - 알레르기 재료 목록
- 개월 수와 생년월일은 양방향으로 자동 동기화되어야 한다.
- 여러 명의 아이 프로필을 관리할 수 있어야 한다.
- 선택된 아이 기준으로 식단과 이력이 분리되어야 한다.

### 5.2 재료 입력

- 재료는 아침 / 점심 / 저녁별로 독립 입력한다.
- 태그 기반 입력을 사용한다.
- 쉼표와 Enter로 다중 입력할 수 있어야 한다.
- 재료 입력은 표준 키 기반으로 정규화되어야 한다.

### 5.3 알레르기 처리

- 알레르기 재료는 메뉴 후보 검색 단계에서 반드시 제외한다.
- 사용자가 알레르기 재료를 입력하면 경고를 표시하고 계산에서 제외한다.
- AI 결과에도 알레르기 재료가 포함되면 안 된다.

### 5.4 식단 추천

- 시스템은 항상 하루 3끼를 생성해야 한다.
- 각 끼니는 해당 끼니의 입력 재료를 우선 사용해야 한다.
- 하루 전체에서 조리법과 주재료가 과도하게 반복되지 않도록 구성해야 한다.
- 메뉴 선택은 규칙 기반 엔진이 먼저 수행해야 한다.
- AI는 메뉴 선택이 아니라 자연어 결과 생성만 담당해야 한다.

### 5.5 결과 표시

각 끼니 결과에는 아래 정보가 포함되어야 한다.

- 메뉴명
- 추천 문구
- 사용 재료
- 부족 재료
- 대체 가능한 재료
- 주의사항
- 조리법 3줄
- 필요 시 fallback 여부

### 5.6 히스토리

- 아이별 최근 식단 이력을 저장해야 한다.
- 저장된 이력은 다시 불러와 볼 수 있어야 한다.
- 오늘 / 최근 / 이전 그룹으로 스캔이 가능해야 한다.

## 6. 비기능 요구사항

### 6.1 사용성

- 1분 이내에 하루 식단 생성이 가능해야 한다.
- 모바일에서 한 손 사용이 가능해야 한다.
- 복잡한 설정 없이 바로 사용할 수 있어야 한다.

### 6.2 안정성

- AI가 실패해도 식단 생성은 중단되면 안 된다.
- fallback 결과를 제공해야 한다.
- 네트워크 오류가 나면 사용자에게 이해 가능한 오류를 보여줘야 한다.

### 6.3 확장성

- 향후 주간 식단으로 확장 가능해야 한다.
- 향후 월령 확장과 장보기 기능을 붙일 수 있어야 한다.
- AI 모델과 프롬프트 정책을 교체할 수 있어야 한다.

## 7. 데이터 저장 정책

### 7.1 서버 DB 저장 대상

- 아이 프로필
- 끼니별 입력 재료
- 하루 식단 결과
- 끼니별 상세 결과
- AI 생성 로그
- 익명 사용자와 사용자 연결 정보

### 7.2 로컬 저장 대상

- 선택한 아이
- 작성 중인 식단 draft
- 익명 세션 복원용 최소 키

### 7.3 source of truth

- 아이 프로필과 식단 결과의 source of truth는 Supabase DB이다.
- 로컬 저장은 UI 편의와 세션 복원에 한정한다.

## 8. 주요 데이터 구조

### 8.1 주요 테이블

- `children`
- `ingredients`
- `menus`
- `meal_inputs`
- `meal_plans`
- `meal_plan_items`
- `ai_generation_logs`
- `anonymous_users`
- `user_identity_links`

### 8.2 저장 원칙

- 재료는 표준 키로 저장한다.
- 하루 식단은 `meal_plans`, 끼니별 결과는 `meal_plan_items`에 저장한다.
- 아래 항목은 항상 저장한다.
  - 부족 재료
  - 대체 재료
  - AI 추천 문구
  - 조리법 요약
  - fallback 여부
  - prompt version

## 9. AI 처리 아키텍처

### 9.1 기본 원칙

- 메뉴 선택은 규칙 기반 추천 엔진이 수행한다.
- AI는 사용자에게 보여줄 자연어 결과만 생성한다.
- AI 결과는 검증 후에만 저장 및 출력한다.
- AI가 실패해도 서비스는 fallback으로 정상 동작한다.

### 9.2 호출 단위

- AI 호출은 **끼니별로 처리한다**.
- 하루 식단 생성 시 OpenAI 요청은 최대 3회 발생한다.
  - 아침
  - 점심
  - 저녁

### 9.3 AI 입력

각 끼니별 OpenAI 요청에는 아래 정보가 포함된다.

- `promptVersion`
- `child.ageMonths`
- `child.allergies`
- `mealType`
- `normalizedInputIngredients`
- `selectedMenu`
- `expectedMissingIngredients`
- `expectedSubstitutes`
- `candidates`
  - 이름
  - 조리 스타일
  - 주 단백질
  - 사용 재료
  - 부족 재료
  - 대체재
  - 식감 메모
  - 주의사항

### 9.4 AI 출력

AI는 구조화된 JSON으로 아래를 반환해야 한다.

- `selectedMenu`
- `recommendation`
- `missingIngredients`
- `missingIngredientExplanation`
- `substitutes`
- `recipe`
- `caution`

### 9.5 검증 규칙

- `selectedMenu`는 미리 선택된 메뉴와 같아야 한다.
- `missingIngredients`는 백엔드 계산값과 같아야 한다.
- 대체재는 허용된 후보 범위 안에서만 작성해야 한다.
- 알레르기 재료와 위험 표현이 포함되면 안 된다.
- 조리법은 3줄이어야 한다.

### 9.6 fallback 정책

아래 경우 fallback을 사용한다.

- OpenAI 설정 없음
- OpenAI 호출 실패
- 구조화 응답 검증 실패
- 안전성 가드 실패

## 10. 후속 확장 방향

- 카카오 로그인 재도입
- 주간 식단
- 장보기 연동
- 추가 통합/E2E 테스트
